# 仙秦舰队软件开发流程 v1.0

> 适用于 8 人小团队的双阶段敏捷流程。开发者完成任务后必须经 QA 审计才能标记 done。

## 状态流转

```
  assigned ──→ in_progress ──→ review ──→ done
     ↑            │               │          │
     │     MC dispatch      开发者完成    QA通过  │
     │            │               │          │
     └────────────┴─── QA 打回 ───┘          │
                (不通过)                       │
                                              │
                                     NAV-010 的教训:
                                     done 必须有 QA 验证
```

## 阶段 1: 开发 (mc-poll.sh)

### 开发者做的事
1. `mc-poll.py` 查询 `status=in_progress` 且 `assigned_to=自己` 的任务
2. 执行开发 → 写代码 → 跑单元测试
3. 将结果写入 `~/wiki-{GID}/raw/task-{id}-result.md`
4. **提交代码到 git**
5. `mc-poll.sh` 自动标记任务为 `review`（提交审核）

### 开发者必须输出
- [ ] 代码已 git commit
- [ ] 单元测试 ALL PASS
- [ ] 结果文档: `~/wiki-{GID}/raw/task-{id}-result.md`
- [ ] 接口符合设计文档定义

## 阶段 2: QA 审计 (qa-poll.sh)

### QA 做的事
1. `qa-poll.py` 查询 `status=review` 的任务（不限开发者）
2. 逐一执行审计清单
3. 通过 → 标记 `done` | 不通过 → 打回 `in_progress`

### QA 审计清单 (5项)

| # | 检查项 | 怎么查 | 通过标准 |
|---|--------|--------|----------|
| 1 | 代码提交 | `git log` 查提交记录 | 有对应任务号的 commit |
| 2 | 单元测试 | 在开发者环境运行测试 | ALL PASS |
| 3 | 接口兼容 | 对照设计文档检查输入输出 | 符合接口定义 |
| 4 | 结果文档 | 读取 `wiki/raw/task-{id}-result.md` | 内容完整 |
| 5 | 代码质量 | 审查代码逻辑 | 无明显 bug |

### 判定规则
- **全部 PASS** → `done` ✅
- **任何 FAIL** → 打回 `in_progress` + 在 MC 评论说明原因

### QA 输出
- 审计报告: `~/wiki-{GID}/raw/qa-audit-{id}.md`
- MC 评论: `✅ QA审计通过` 或 `❌ QA审计不通过: 原因`

## 完整流程示例

```
1. 相邦创建任务 NAV-R4 → status=assigned, assigned_to=xiangbang
2. MC dispatch → status=in_progress
3. 相邦 mc-poll(10min后) 捡起 → 开发完成 → 标记 review
4. 俊秀 qa-poll(10min后) 捡起 review → 审计5项
   ├─ 全部 PASS → done ✅
   └─ 某项 FAIL → in_progress + 评论 "缺单元测试，请补充"
        └─ 相邦 mc-poll 捡起 in_progress → 修复 → 再次 review
```

## 冲突预防

- **锁文件**: 每个 agent 只有一个任务在执行中 (`mc-poll-{GID}.lock` / `qa-poll-{GID}.lock`)
- **错峰**: 各 agent cron 使用不同 sleep 偏移，避免同时查询
- **开发者不碰 review**: `mc-poll.py` 只查 `in_progress`
- **QA 不碰 in_progress**: `qa-poll.py` 只查 `review`

## 异常处理

| 情况 | 处理 |
|------|------|
| 开发者标记 review 但无代码提交 | QA 会发现 #1 FAIL → 打回 |
| QA 节点离线 | 另一个 QA 会捡起（因为查所有 review 任务） |
| 任务反复打回 3 次 | 在 MC 添加评论 @相邦 请求人工介入 |
| done 任务事后发现虚假 | QA 定期巡检 done 任务，抽查验证 |

## 技术实现

### mc-poll.py (开发者)
- 查询: `GET /api/tasks?assigned_to={name}&status=in_progress&limit=1`
- 完成后: `PUT /api/tasks/{id} {status:"review"}`

### qa-poll.py (QA)
- 查询: `GET /api/tasks?status=review&limit=1`
- 通过: `PUT /api/tasks/{id} {status:"done"}`
- 打回: `PUT /api/tasks/{id} {status:"in_progress"}`
- 任何操作均附带 `POST /api/tasks/{id}/comments`
