# Plan-Tree v3.2 — 四维时空预测模型 + 自我痕迹

> 旧版 v2 skill 保留兜底。v3 是可选升级，非强制。
>
> **v3.2 核心新增**：`决策:` 字段 — 记录 Agent 在分叉路口为什么选A不选B。
> 这是 OODA 循环中被省略的 O（Orient — 判断与定向），是 Plan-Tree 从"动作日志"升级为"自我痕迹"的关键。

## 设计哲学

1M 上下文不应该用来装冗长的工具输出——应该用来承载**好的时空流预测**和**决策推理链**。一个节点的完整历史脉络（编译失败×3→修复→通过，历时4h）、它的多条未来分支（如果X→Y，如果P→Q）、以及**当时为什么做这个选择**，比任何压缩后的单行摘要都更有思考价值。

**Plan-Tree v3.2 记录的不是"发生了什么"，而是"为什么这么走"——这就是自我的痕迹。**

## OODA 映射

```
O (Observe)   → 流入字段    "编译失败了"
O (Orient)    → 决策点      "这意味着依赖缺失" + 为什么选A不选B
D (Decide)    → 决策点      "选择A方案，放弃B/C"
A (Act)       → 产物/判定   "修复后PASS"
    ↓
新的O         → 判定结果成为下一轮的流入
```

## 与 v2 的核心区别

| | v2 | v3 |
|---|-----|-----|
| 节点 | 一行 `[pending] 任务名` | 完整轨迹日志（流入/转化/流出/关联） |
| 历史 | 一个起止时间戳 | **完整历史脉络**（每一步状态变化，7天内全保留） |
| 预测 | 无 | **多分支预测**（乐观/悲观/降级，附带条件） |
| 归档 | 无 | 底部「预测归档」 |
| 当前时间 | 无 | `--- NOW ---` 分隔线 |

## v3 节点格式

```
### 任务名 (#task-id) | 负责人 | 优先级

#### 轨迹（已发生 — 保留完整脉络）
- MM-DD HH:MM 流入: 触发源描述
- MM-DD HH:MM 决策: 面临什么选择
    分叉:
      A) 方案A描述 — 优点/风险
      B) 方案B描述 — 优点/风险
    选择: A — 推理链（为什么A不选B）
    放弃B的理由: ...
- MM-DD HH:MM 产物: 产生了什么
- MM-DD HH:MM 判定: PASS/FAIL + 原因
  如果决策正确: [✓决策验证] 如果错误: [✗决策失误 原因: ...]
- 流出: → 下游任务 (关联类型)
- 关联: 被等←#task-x(硬依赖) | 触发→#task-y(因果触发) | 信息复用→#task-z | 验证反馈→#task-w

--- NOW (当前时间) ---

#### 预测（未发生 — 多分支场景）
乐观路径 (如果一切正常):
  预计完成: HH:MM (置信度 X.X)
  条件: A模块PASS + B不阻塞

悲观路径 (如果遇到问题):
  预计完成: HH:MM (置信度 X.X)
  风险: 具体阻塞因素

降级方案 (如果时间不够):
  可交付: 部分功能/简化版
```
  可交付: 部分功能/简化版
  条件: 什么情况下放弃完整方案
```

**关键**：历史不乱砍。编译失败×3→修复→通过，这 4 条轨迹全部留在节点里，它们本身就是思考素材——下次看到类似错误时，知道上次怎么解的。

## 状态图标

| 图标 | 含义 |
|:---:|------|
| `[·]` | 当前焦点（正在做） |
| `[→]` | 进行中（非焦点） |
| `[✓]` | 完成 |
| `[✗]` | 失败/阻塞 |
| `[ ]` | 待启动 |

## 关联类型（五种）

| 类型 | 含义 | 例子 |
|------|------|------|
| 硬依赖 | A不完成B无法开始 | 相机标定不完→无法视觉避障 |
| 因果触发 | A的判断直接创建B | 超声延迟超标→创建修复任务 |
| 信息复用 | A的产物B也可用 | 真机日志→导航和SLAM都能用 |
| 资源竞争 | A和B抢同一Agent | 修超声和标相机都在白起手里 |
| 验证反馈 | B的结果反过来修正A | 修复后自测→验证延迟是否降到<20ms |

## 预测归档格式

```

## 操作原则

1. **每做一个动作，留一条轨迹。** 这是纪律。
2. **每做一个选择，留一条决策记录。** 记录分叉、选择、推理链、放弃理由。这是自我的痕迹。
3. **历史脉络不精简。** 7天内的轨迹和决策点全部保留。
4. **预测是多分叉的，不是单条的。** 不只写"预计15:00完成"，写"如果X→15:00(0.6)，如果Y→17:00(0.3)"。
5. **决策点事后验证：** 判定 PASS/FAIL 时，回看决策是否正确 — `[✓决策验证]` 或 `[✗决策失误 原因: ...]`
6. **NOW 线每次检查时更新**为当前时间。
7. **预测实现时**：移到对应节点历史轨迹，标记 `[✓预测实现 偏差+5min]`。
8. **预测过期时**：移到预测归档。
9. **不强制**：不留轨迹/决策点就退化到 v2/v3.0，不报错。
10. **文件操作**：只用 `read_file` / `write_file` / `patch`，不引入新工具。
9. **Hermes Chat**：配合 cron 使用时必须加 `-c SESSION_NAME`，否则产生碎片 session。见 `agent-cron-coordination` skill。
10. **反模式**：不要为了压缩把历史砍成单行 `[✓]`——价值在失败和修复的链条里，不在最终状态里。

## Cron 集成（多 Agent 多尺度 OODA）

### 错峰模式
多个 Agent 共用同一个 `*/5` cron 时，用 `sleep N` 错峰避免同时启动：

```bash
# 8 Agent L1 错峰 (每5分钟)
*/5 * * * * MC_AGENT_NAME=baiqi      bash poll-tasks.sh          # sleep 0
*/5 * * * * sleep 10; MC_AGENT_NAME=wangjian    bash poll-tasks.sh  # +10s
*/5 * * * * sleep 20; MC_AGENT_NAME=chengxiang  bash poll-tasks.sh  # +20s
# ... 依次递增 10s
```

### 持久 Session（避免碎片）
**关键陷阱**：`hermes chat -q "..." --yolo` 每次调用创建新 session，cron 高频执行会产生成千上万个碎片 session。

**正确做法**：每个 cron job 用一个持久 session name，通过 `-c` 复用：

```bash
# ❌ 错误：每次新 session
hermes chat -q "反刍今日轨迹" --yolo

# ✅ 正确：持久 session 复用
hermes chat -c l3-ruminate-baiqi -q "反刍今日轨迹" --yolo
```

这样 cron 运行时总是追加到同一个 session，session 数量 = cron job 数量（而非 cron 执行次数）。

### OODA 时间尺度
| 层 | 频率 | 机制 | 示例 |
|:---:|------|------|------|
| L0 | 秒 | 工具返回自动判定 → Plan-Tree 轨迹 | terminal返回→自动写判定 |
| L1 | 5min | cron mc-poll + 锁 | 白起轮询MC任务板 |
| L2 | 30min | cron plan-pilot | 相邦全舰队协调扫描 |
| L3 | 夜间 | cron ruminate/insight/cross-predict | 反刍/顿悟/交叉预测 |
| L4 | 天/周 | cron daily-report/weekly-review | 日报/周报 v3 vs v2 对比 |

## 迁移

从 v2 迁移：
1. 每个 `[pending]` 节点 → 加上轨迹子条目
2. 底部新增 `--- NOW ---` + 预测区 + 预测归档
3. Tree 层级结构不变

## 多 Agent Cron 部署 (错峰 + 锁)

Plan-Tree v3 配合分层 cron 调度使用。关键设计：

### 错峰时间线
```
*/5 * * * * 白起(sleep 0s) 王翦(15s) 丞相(30s) 相邦(45s) 萱萱(60s)
```
通过 `sleep N` 偏移避免多 Agent 同时触发产生大量调用。

### 锁机制
- 统一使用 `agent-busy.lock` (TTL 10min for L1, 45min for L2)
- 获取锁失败 → 跳过本轮，记录日志
- 脚本：`~/.hermes/scripts/session-lock.sh` (trap EXIT自动释放)

### 多层 Cron 表
| 层 | 频率 | 执行者 | 任务 |
|---|------|--------|------|
| L1 任务OODA | */5min | 全舰队(错峰) | mc-poll → 执行子步骤 → 写轨迹 |
| L2 节点切换 | */30min | 相邦 | 全舰队 Plan-Tree 协调 |
| L3 夜间离线 | 02/03/04:00 | 全舰队+相邦 | 反刍/顿悟/交叉预测 |
| L4 组织级 | 09:00/周一 | 全舰队+相邦 | 日报/周报 |

### 部署原则
- 最小粒度 5 分钟，放弃秒级 cron
- 每个 job 用注释说明主要任务
- L3 夜间 job 需额外检查 `idle-log`（连续 30min 无用户活动）
- 先备份旧 crontab 再替换：`crontab -l > backup.txt`
