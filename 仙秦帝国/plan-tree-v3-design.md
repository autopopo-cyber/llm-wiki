# Plan-Tree v3 设计文档

> 版本: 1.0 | 2026-04-27 | 萱萱 设计
> 将 Plan-Tree 从静态任务列表升级为四维时空预测模型
> 核心原则：纯 Markdown、轻量、Agent 视角、不引入新工具

---

## 一、它长什么样

以白起的 NAV_DOG 闭环为例。**当前时间 = 04-27 14:00**，白起打开自己的 plan-tree 看到：

```markdown

## ENSURE_CONTINUATION
- [✓] 自测全绿 | 04-27 08:03 | 耗时12s
- [✓] DB备份   | 04-27 08:05 | rsync → /backup

## EXPAND_CAPABILITIES

  - 04-25 09:00→04-26 02:00 | 15h | 产物: docker-compose.yml + 仿真配置
  - 关联: 触发→相机标定, 触发→避障回路
```

## 二、它会挤占大量上下文吗

**不会。** 设计原则：Agent 每次只加载自己关心的切片。

| 加载策略 | 内容 | 预估 token |
|---------|------|:---:|
| **默认加载** | 当前焦点节点 + 直接关联节点(1跳) + 24h预测 | ~300-500 |
| **展开加载** | 某条因果链的全部节点(如 NAV_DOG 完整链路) | ~800-1200 |
| **全局加载** | 整个 Tree 的结构骨架(无轨迹细节) | ~400-600 |
| **全量加载** | 所有节点 + 所有轨迹 + 预测归档 | ~3000-5000 |

对比当前 Plan-Tree v2（白起约 1200 token），v3 默认加载只多了 200 token，但信息密度翻倍。

**关键**：Agent 不需要从头读 Plan-Tree。它的 Tree 层级是骨架，流入/流出/关联是按需展开的。就像你不会一次性读完整本 wiki——你读当前相关的那几页。

## 三、压缩信息的手段

| 手段 | 例子 | 压缩比 |
|------|------|:---:|
| **缩略键值对** | `04-27 10:30 产物: 真机日志 (超声延迟 35ms)` → 一行 | 3:1 |
| **时间戳精简** | 同日省略日期 `10:30` 代替 `2026-04-27 10:30:00` | 2:1 |
| **关联短ID** | `触发→#fix-02` 代替 `触发了任务"超声采样率修复"(#fix-02)` | 3:1 |
| **状态图标** | `[→]`=进行中 `[·]`=当前焦点 `[✓]`=完成 `[✗]`=失败 | 5:1 |
| **预测模板化** | `预计 1.5h 置信度 0.6 风险: 联动测试` | 3:1 |
| **引用代替复制** | `被等←#cam-01(硬依赖)` 不复制 #cam-01 的全部内容 | 10:1 |
| **TTL 自动清理** | 3天前的预测归档自动 `→ 预测历史库` (详见 §六) | ∞:∞ |

这套压缩下来，一个节点的完整轨迹 ≈ 5-8 行 Markdown，约 80-150 token。10 个活跃节点 ≈ 1000 token——在 Agent 上下文中占比不到 1%。

## 四、方便修改的手段

Plan-Tree v3 是纯文本。任何文本编辑器都能改。但为了提高效率：

**手动修改**：
- 添加轨迹：在对应节点下新加一行 `- 04-27 14:30 判定: ...`
- 更新预测：修改 `预测:` 行的数值和置信度
- 完成节点：`[·]` → `[✓]`，加一行完成时间

**LLM 辅助修改**（日常操作的主要方式）：
```
// Agent 在每次工具调用后，自己在 Plan-Tree 加一行轨迹
// 不是靠 cron，是动作驱动的
操作: 终端返回 → Agent 在 Plan-Tree 写入 "产物: xxx.log"
操作: 判定结果 → Agent 写入 "判定: PASS/FAIL + 原因"
操作: 触发新任务 → Agent 写入 "关联: 触发→#new-task"
```

**不需要新工具。** 当前的 `read_file` / `write_file` / `patch` 完全够用。唯一需要的约定是：**Agent 每完成一个动作，必须在 Plan-Tree 留下一条轨迹。** 这是纪律，不是技术。

## 五、"当前时间"：区分预测和现实

用一条 `--- NOW ---` 分隔线。线上是**已发生的**（历史轨迹），线下是**尚未发生的**（预测）。

```
历史轨迹（已发生）
  04-27 10:30 产物: 真机日志
  04-27 10:35 判定: FAIL
  04-27 13:00 产物: 修改驱动参数
  ...
--- NOW (04-27 14:00) ---
预测（未发生）
  预计 04-27 15:00 编译通过 (置信度 0.6)
  如果延迟 → 降级方案: 纯超声模式
```

**"当前时间"是移动的。** Agent 每次检查 Plan-Tree 时，把 `NOW` 更新为当前时间。比 NOW 早的事件就是历史，比 NOW 晚的就是预测。

**预测实现时**：对应的预测行从"预测区"移到对应节点的历史轨迹中，标记 `[✓预测实现]`。

**预测过期时**：移到 `预测归档`，标记原因。

## 六、未实现的预测：不丢，归档

不丢。预测的准确性是**校准 LLM 预测能力**的数据源。分三层：

| 状态 | 位置 | 格式 |
|------|------|------|
| **活跃预测** | Plan-Tree 底部「预测区」 | `预计 15:00 完成 置信度 0.6` |
| **已实现预测** | 对应节点的历史轨迹 | `04-27 15:05 完成 [✓预测实现，偏差+5min]` |
| **已过期预测** | Plan-Tree 底部「预测归档」 | `04-26预测: 12:00完成 → 实际未完成，原因:超声延迟 → 触发#fix-02` |

**归档保留 7 天**。7 天后的预测归档 → 移动到 `~/wiki-2/plan-tree-archive/`，永不过期，但不在 Plan-Tree 上下文里占位。

**归档的价值**：当你回头看一个 Agent 的预测准确率——"过去 30 天，白起的任务完成预测偏差中位数是多少？"——这些数据都在。这是组织自我校准的基础。

## 七、v2 → v3 迁移：改什么

| 当前 v2 | v3 | 改动 |
|---------|-----|------|
| `- [pending] 避障回路验证` | 保留 `[·]`，新增子条目(流入/转化/流出/关联) | 加字段 |
| 时间戳只在节点头 | 每个子条目有时间戳 | 加时间 |
| 无预测 | 底部「预测区」 | 新区域 |
| 无 NOW 线 | `--- NOW ---` 分隔线 | 一行 |
| 无归档 | 底部「预测归档」 | 新区域 |

**本质改动就一个**：每个节点从一行，变成带时间轴的微型日志。Tree 的层级结构不变。

---

## 八、约束

1. **不引入新工具** — 纯 Markdown + 现有文件操作
2. **不强制** — Agent 可以不写轨迹，不写就没预测能力（退化到 v2）
3. **不强一致** — 不同 Agent 的 Plan-Tree 独立，MC 做聚合
4. **可容错** — 写错了随时文本编辑修正，不依赖数据库事务
5. **可恢复** — 文件在 Git 里，删了 `git checkout`
