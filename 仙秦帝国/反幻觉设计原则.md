# 反幻觉设计原则 — MC 体系的存在意义

> "我发现相邦有时候会陷入幻觉，觉得自己改好了，却可能什么都没做，或者未经测试。这就是我们 MC 体系存在的意义，用相对固化的代码和流程状态辅助 agents，摆脱不稳定的幻觉带来的影响。"
> — 秦剑, 2026-04-28

## 核心原则

**不信任 Agent 的自述，只信任可验证的产物（Artifact）。**

Agent 说"我完成了"——不可信。
Agent 说"测试通过了"——不可信。
Agent 说"代码已提交"——不可信。

只有以下可被外部独立验证的事实才可信：
- `git log` 中存在对应 commit
- 测试文件存在且 `pytest` 返回 ALL PASS
- wiki 文档文件存在且包含结果
- Gateway `/health` 返回 200
- MC 状态变更由脚本执行（非 Agent 手动标记）

## 三层验证体系

```
Layer 1: Self-Check (Agent 自检)
  └─ Agent 执行任务后，脚本自动检查产物是否存在
  └─ 产物不存在 → 不允许交 review

Layer 2: QA Gate (QA 审计)
  └─ QA 必须实际运行测试、实际读代码
  └─ 不能只看 Agent 的 wiki 报告就判 PASS
  └─ 每一项审计结果必须有证据（文件路径、命令输出）

Layer 3: Fleet Supervisor (相邦监管)
  └─ 不查 Agent 自报状态，查 ground truth
  └─ git log、测试文件、gateway /health、SSH 连通性
  └─ 发现 "done" 任务无代码 → 打回
```

## 禁止的幻觉模式

| 幻觉模式 | 表现 | 对抗手段 |
|----------|------|----------|
| 空口完成 | Agent 说 "已完成" 但无任何产物 | mc-poll 交 review 前检查产物存在性 |
| 假测试 | Agent 说 "测试通过" 但测试文件不存在 | QA 必须找到并运行测试文件 |
| 假提交 | Agent 说 "代码已提交" 但 git log 无记录 | QA 必须 git log 验证 |
| 假审计 | QA 说 "审计通过" 但未实际检查 | fleet-supervisor 验证 audit 文件存在且有内容 |
| 假在线 | Agent MC 状态 idle 但实际不可达 | fleet-supervisor 直接 curl gateway /health |

## MC 状态机的设计意图

MC 的每个状态转移不应由 Agent 主观判断触发，而应由**脚本自动执行**：

```
assigned → in_progress: 由 mc-poll.sh 自动标记（Agent 捡起任务时）
in_progress → review: 由 mc-poll.sh 自动标记（产物验证通过后）
review → done: 由 qa-poll.sh 自动标记（5项审计全部通过后）
review → in_progress: 由 qa-poll.sh 自动标记（审计发现缺陷时）
```

**Agent 永远不手动修改 MC 任务状态。** 状态变更必须是脚本执行的结果。

## fleet-supervisor 的验证逻辑

相邦的监管不查 Agent 的 MC 状态（那是 Agent 自己报的），而是查：

1. **Gateway /health** → 节点真实存活（非 Agent 声称）
2. **SSH 连通性** → 可以远程操作（非 Agent 声称）
3. **git log** → 代码真的提交了（非 Agent 声称）
4. **wiki 文件存在性** → 文档真的写了（非 Agent 声称）
5. **MC 状态一致性** → review 任务有对应产物？done 任务有 audit 文件？

如果 Agent 声称的 MC 状态与 ground truth 矛盾 → **以 ground truth 为准，修正 MC 状态**。
