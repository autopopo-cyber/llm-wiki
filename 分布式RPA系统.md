# 万能分布式 RPA 系统

## 项目定位

端侧视频流分析 + Playbook 执行的分布式机器人协作系统。

## 核心特性

| 特性 | 描述 |
|------|------|
| **端侧推理** | 在机器人端侧分析视频流，低延迟决策 |
| **Playbook 执行** | 预写序列 + VLM 实时远控，两种模式切换 |
| **多机协作** | 同时控制多个机器人/机器狗协同工作 |
| **万能** | 不绑定特定机器人平台，通用 RPA 框架 |
| **动态路径分发** | 路线和 Plan 由远端动态下发，非固化 |

## 算力分层架构

RPA 的核心难题是算力分配。按帧率需求从低到高分四层，每层部署位置不同：

| 层级 | 帧率 | 部署位置 | 任务 | ABot-Claw 对应 |
|------|------|---------|------|---------------|
| L4 规划决策 | **超低帧** (0.1-1Hz) | 云端 | 任务分解、全局规划、VLM推理 | OpenClaw 层（高层决策） |
| L3 视觉判断 | **低帧** (1-5Hz) | 机载 GPU (Jetson) | VLM 场景理解、语义分割 | 共享服务层（YOLO-World/AnyGrasp） |
| L2 视觉分类 | **高帧** (10-30Hz) | 机载 GPU (Jetson) | YOLO 检测/分割、实时避障感知 | 共享服务层（检测） |
| L1 运动控制 | **超高频** (100-1000Hz) | 机器人板载 | RL 平衡、步态生成、关节力矩 | 机器人层（ROS/ROS2） |

**关键洞察：** 这四层与 ABot-Claw 的三层架构完全对齐——OpenClaw=L4、共享服务=L2+L3、机器人层=L1。

而 **Hermes（Claw）= L4↔L3 的桥梁**：
- 云端大模型/VLM 做不了实时决策（太慢）
- 端侧 YOLO/RL 做不了高层推理（太笨）
- 中间需要一个 **Agent** 接收云端意图、编排端侧行为、上报执行结果
- ABot-Claw 的 "Claw"（利爪/驾驭）就是这个角色——**抓取云端指令，驾驭端侧行为**
- Hermes 天然就是这个 Claw

```
    云端 LLM/VLM                    ← "去哪、做什么"
         │
    ┌────┴────┐
    │  Hermes  │  ← Claw = 云端与端侧之间的 Agent
    │ (Claw)  │     接收意图 → 编排 Playbook → 调度端侧 → 上报结果
    └────┬────┘
         │
    机载 GPU (Jetson)                ← "看到了什么、怎么避开"
    YOLO + VLM(轻量)
         │
    机器人板载                        ← "怎么迈步、怎么站稳"
    RL 运控 (1000Hz)
```

## VLA 的失败与 Agent 的胜利

### VLA 路线为什么失败（2024-25）

VLA（Vision-Language-Action）试图让 VLM 直接输出 Action token（关节角度、速度指令等），但失败了：

| 问题 | 原因 |
|------|------|
| **Token 吞吐量不足** | VLM 生成 token 速度 ~10-30 tok/s，机器人 1000Hz 控制回路需要每秒 1000+ 条指令，根本供不上 |
| **延迟不可接受** | VLM 单次推理 100ms-1s，RL 运控需要 <10ms 响应，差 1-2 个数量级 |
| **精度不够** | VLM 输出的是离散 token，无法精确表达连续控制量 |
| **训练数据瓶颈** | VLA 需要视觉-语言-动作三元组数据，极度稀缺 |

### RL 大放异彩（2024-25）

同一时期，RL 在运控层取得了突破性进展：
- Unitree G1 跳舞、打功夫
- Boston Dynamics 体操、跑酷
- 四足全地形鲁棒行走

**RL 成功的原因：** 它不试图"理解"世界，只管"怎么站稳、怎么迈步"——这是一个定义明确的 MDP 问题，仿真数据可以无限生成，Sim2Real 迁移技术成熟。

### Agent 桥接了 VLM 和 RL（2026）

**Hermes/Agent 是缺失的那一层：**

```
VLM "看到目标了" ──✗──→ 直接输出 Action（VLA 路线，失败）
VLM "看到目标了" ──→ Agent 翻译成 Playbook ──→ RL 执行 Action（成功）
```

| 路线 | 链路 | 问题 |
|------|------|------|
| VLA（直接） | VLM → Action token | 太慢、太少、太粗 |
| **Agent（间接）** | **VLM → Agent → Playbook → RL** | **每层各司其职，帧率匹配** |

**这才是 VLA 真正的实现方式——不是让大模型直接输出 Action，而是通过 Agent 间接实现 Action。**

ABot-Claw 就是这个路线的完整验证：
- 云端 VLM 理解场景 → Claw(Agent) 编排行为 → 端侧执行 → RL 运控
- 全流程从云到 CAN 总线打通
- 响应速度可接受（因为每层只做自己帧率匹配的事）
- 还能控制机械臂——几乎任意控制都实现了

**"VLA 实现了，虽然不是直接输出。"**

## 架构原则：同一模式，不同尺度

VLA→Agent 的转型揭示了一个通用架构原则，它不仅适用于机器人，也适用于 Agent 自身：

| 层级 | 机器人世界 | Hermes 自身 | 职责 |
|------|-----------|------------|------|
| 理解层 | Cloud LLM | GLM-5.1 | 只做判断，不做执行 |
| Agent 层 | Claw/Hermes | DeepSeek v3.2 + 工具 | 把意图翻译成可执行步骤 |
| 执行层 | RL/OS/Can总线 | 终端/代码/浏览器 | 精确执行，不管为什么 |

**核心原则：理解层永远不直接输出动作。** VLA 失败的原因就是让理解层跳过 Agent 层直接到执行层。无论尺度大小——从云端到 Can 总线，从对话到终端命令——都是这三层。

## ABot-Claw 的关键启示

重读 ABot-Claw 后，RPA 系统获得了三个核心设计参考：

### 1. 路线动态分发 → 协作基础

当前导航系统的路线是固化的（预录或本地计算）。ABot-Claw 证明：**路线和 Plan 由远端动态分发，机器人才具备协作基础**。

- 高德地图 API 生成全局路径 → 云端转换坐标 → 下发给端侧导航系统
- 端侧只负责"怎么走"，云端负责"去哪、做什么"
- 这正好是 RPA Playbook 的一个 Step：`navigate_to(point_A)`

### 2. 任务上下文跨机移交 → 容错关键

ABot-Claw 的"一个运行时、多智能体共生"：

> 如果一台四足机器人电量不足，另一台可以接手继续执行，不需要重新理解任务、重新规划路径。

这对 RPA 意味着：**任务上下文不能只存在于单机**。Playbook 的执行状态、当前步骤、环境快照必须上云，才能实现跨机移交。

### 3. Map as Memory → 共享世界记忆

ABot-Claw 四层视觉空间记忆：

| 层级 | 内容 | RPA 价值 |
|------|------|---------|
| BlockLayer（区域层） | 室内房间/室外街区 | 粗粒度任务分解 |
| RoadLayer（路网层） | 交叉口、门洞、通道 | 路径规划硬约束 |
| FunctionLayer（功能层） | 休息区、厨房、电梯厅 | 语义意图→导航目标 |
| ObjectLayer（对象层） | 特定物品/商铺位置 | "最后一米"精确定位 |

**核心洞察：** 机器人 A 探索过的环境可以零成本迁移给机器人 B。RPA 系统应维护一份共享的空间语义记忆，新机器人上线即插即用。

### 4. 闭环反思 → 自我纠错

ABot-Claw 的 Self-Reflector：任务失败时生成结构化诊断反馈，触发重新规划。对 RPA 来说，Playbook 不应是线性执行，每个 Step 完成后需要：

- ✅ 成功 → 下一步
- ⚠️ 部分成功 → 微调重试
- ❌ 失败 → 生成诊断 → 重规划或换机器人

### 5. 云端大脑 + 边缘响应

ABot-Claw 的两级架构：

| 层级 | 职责 | 延迟 |
|------|------|------|
| 云端（L3/L4） | 任务分解、全局规划、VLM推理 | 100ms-1s |
| 边缘（L1/L2） | 避障、SLAM、实时控制 | <20ms |

RPA 系统天然适配这个模型：VLM 远控走云端，避障和导航走边缘，网络断开时边缘侧仍能保底运行。

## 与导航项目的关系

| 导航系统 | RPA 系统 |
|---------|---------|
| 机器狗怎么走 | 机器狗做什么 |
| 底层能力（SLAM/避障/路径跟踪） | 上层调度（任务编排/协作/远程控制） |
| 单机自治 | 多机协同 |
| 被动接收路径 | 主动请求/接收路径 |
| 固化路线 | **动态路线分发** |

**关键进化：** 导航系统原来是"知道怎么走但路线固定"，RPA 系统让它变成"路线也能动态下发"。这正是 ABot-Claw 已实现的模式。

## 技术选型

| 模块 | 选型 | 理由 |
|------|------|------|
| 端侧 VLM | Qwen2.5-VL（云端推理） | 算力有限，VLM 推理放云端更现实 |
| 通信协议 | gRPC（控制流）+ MQTT（状态流） | gRPC 低延迟双向流，MQTT 轻量广播 |
| Playbook 格式 | YAML + Jinja2 模板 | 类 Ansible，非开发者友好，模板支持参数化 |
| 任务编排 | 行为树 | 比 FSM 更灵活，天然支持并行/条件/重试 |
| 共享记忆 | 向量数据库 + 空间索引 | 类 ABot-Claw 的混合检索（几何+语义） |
