# 四足导航开发路线

> 基于 Marathongo 项目分析，明确机器狗室内外长距离导航系统的开发方向。
> **核心原则：仿真优先，真机解耦。所有算法在物理引擎中闭环验证，机器狗晃动在仿真中模拟，最小化真机参与。**

## 架构决策：vo_navigation 路线

### 为什么选 vo_navigation 而非 local_planner

| 特征 | 地图导航模式 | local_planner | vo_navigation |
|------|------------|---------------|---------------|
| 全局路径 | 地图API生成 | ✅ smoothed_path | ✅ free_path |
| 核心逻辑 | 跟着走，遇障绕开，绕完回来 | 20条候选选最优 | **跟着走，遇障绕开，绕完回来** |
| 避障方式 | 横向偏移绕行 | 多轨迹采样+Dijkstra | **切弧横向偏移** |
| 路径跟踪 | 最近点追踪 | parallel_score | **最近点追踪** |
| 复杂度 | 简单 | 高（赛车线选择） | **简单** |

**核心判断：** local_planner 是半马竞速产物——在20条候选赛车线中选最优，关心"走哪条线最快"。地图导航本质是通勤——路线已定，跟着走，遇障绕开，绕完回来。这就是 vo_navigation 的逻辑。

## 核心开发原则：仿真优先，真机解耦

**所有导航-避障-运控算法先在仿真引擎中闭环完成开发和验证，再移植到真机。**

**原因：**
- 真机测试成本高（电池、场地、机器狗损耗）
- 边缘状态/危险场景在仿真中可安全复现（冲出赛道、盲区碰撞、两侧夹击）
- 算法迭代速度：仿真秒级 vs 真机小时级
- 机器狗晃动、传感器噪声可在仿真中精确控制

**双引擎仿真策略：**

| 仿真引擎 | 验证范围 | 物理精度 | 场景规模 | 运行频率 |
|---------|---------|---------|---------|---------|
| **Gazebo Ignition（长距离导航仿真）** | 宏观路径跟踪、地图对接、坐标转换、室内外切换 | 中等（简化地形+道路网络） | 公里级（真实 DEM/地图导入） | 10-20Hz |
| **MuJoCo（短距离避障+复杂地面仿真）** | 微观避障算法、切弧逻辑、边缘约束、四足晃动响应 | **高**（精确接触动力学+传感器模拟） | 百米级（人工搭建场景） | 100-200Hz |

**两个引擎不需要同步运行。** 长距离导航验证路线跟踪逻辑，短距离避障验证局部运动控制，各自独立闭环，最后在真机上集成。

## 系统架构

### 三层解耦

用户明确提出**导航-避障-运控三层解耦**，这比 Marathongo 的"导航避控一体"更清晰：

| 层级 | 职责 | 输入 | 输出 | 类比 |
|------|------|------|------|------|
| **导航层** | 路径规划、路径跟踪、目标管理 | 地图 waypoints、当前位姿 | 前视目标点、跟踪误差 | "往哪走" |
| **避障层** | 障碍物检测、可行域计算、绕行决策 | LiDAR/视觉、导航层目标 | 修正后的 cmd_vel | "能不能直走" |
| **运控层** | 步态生成、平衡控制、关节力矩输出 | cmd_vel (vx,vy,ω) | 12关节力矩 | "怎么迈步" |

**三层之间只通过 `cmd_vel` 和状态信号交互，彼此不感知内部逻辑。**

Marathongo 的 `local_planner` 和 `vo_navigation` 其实混合了导航+避障两层，需要拆分开：
- `path_process_node` → **导航层**（最近点追踪、前视目标点）
- `vo_navigation` → **避障层**（切弧绕行、可行域约束）
- 机器狗板载 RL → **运控层**（与 Marathongo 相同）

### 仿真闭环版（Phase 1-4）

```
┌──────────────────────────────────────────────────────────────────────┐
│                        开发环境（Server A/B）                         │
│                                                                      │
│  高德API → waypoints → 坐标转换 → Python脚本/日志输出（文本化调试）    │
│       │                                                              │
│       │ 方式A：长距离导航仿真（Gazebo Ignition）                      │
│       │   导入真实 DEM/道路网络 → 加载四足URDF → 导航层验证           │
│       │   场景：校园/街道公里级路径跟踪                               │
│       │                                                              │
│       │ 方式B：短距离避障仿真（MuJoCo）                              │
│       │   精确接触动力学 + 四足晃动模拟 + LiDAR/Camera噪声注入        │
│       │   场景：边缘约束/障碍物绕行/复杂地面                          │
│       │                                                              │
│       ▼                                                              │
│  ┌───────────────────────────────────────────────────────────────┐   │
│  │  导航层：path_process_node（最近点追踪、前视目标点）            │   │
│  │  避障层：vo_navigation（梯形检测、可行域约束、切弧绕行）        │   │
│  │  运控层：四足RL运控模型（MuJoCo中模拟 cmd_vel → 步态响应）      │   │
│  │  传感器：LiDAR（带晃动）+ Camera（带晃动）+ IMU               │   │
│  └───────────────────────────────────────────────────────────────┘   │
│       │                                                              │
│       │ 文本化输出：cmd_vel序列、跟踪误差、碰撞检测结果、执行日志      │
│       ▼                                                              │
│  Hermes 分析 → 调参 → 重跑 → 闭环迭代                                │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

**APP后端简化：** 先不做安卓APP，用**文本化接口**替代：
- 高德API → waypoints 输出为 JSON/CSV
- 导航结果 → 终端日志 + 可视化轨迹图
- 调试接口 → ROS2 topic echo + Python脚本调参
- AI闭环 → Hermes 读取日志 → 分析 → 建议参数调整 → 重跑

### 真机集成版（Phase 5）

仿真验证通过后，才移植到宇树 A2：
- 导航层+避障层代码 ROS2 移植
- 传感器标定（LiDAR、Camera、IMU）
- 运控层直接使用 A2 板载 RL（无需重新训练）
- 校园/园区实机测试

### 协作版（RPA 系统对接后）

```
┌─ 云端 RPA 调度中心 ─────────────────────────────────┐
│                                                      │
│  高德API → 路径生成 → 坐标转换 → 动态下发给各机器狗    │
│  Playbook → 任务分解 → 路由分配 → 上下文管理          │
│  共享记忆（Map as Memory）← 各狗回传观测              │
│  VLM 远控 → 视频流分析 → 实时指令下发                 │
│                                                      │
└──────────┬──────────────┬──────────────┬─────────────┘
           │              │              │
     ┌─────┴─────┐  ┌────┴─────┐  ┌────┴─────┐
     │  导航系统  │  │  导航系统  │  │  导航系统  │
     │  机器狗A   │  │  机器狗B   │  │  机器狗C   │
     └───────────┘  └───────────┘  └───────────┘
```

**关键进化：** 路线从"本地固化"变为"远端动态分发"，这正是 ABot-Claw 已验证的模式。
单机导航是基础能力，RPA 系统在其上加动态路径分发 + 任务上下文移交 + 共享记忆。

#### 边缘约束测试策略

**也庄半马事故：机器人冲出赛道**。根源是避障算法没有"可行域约束"——遇到边缘障碍时往外侧绕，绕出了赛道。

**MUJoCo 测试场景设计：**

```
┌───────────────────────────────────────┐
│  不可行域（建筑物/边界外）         │
│    ════════════════════════════════════│
│  ┌───────────────────────────────────┐    │
│  │  可行域（道路本体）              │    │
│  │   • · → ─────────────────────→   │    │
│  │  │ ───────────────────────────────────┘    │
│  │  ┌───────────────────────────────────┐  │
│  │  │  障碍物（box）              │  │
│  │  │  ●·····················→ │  │
│  │  │  ·  必须往左侧还能行使的空间  · │  │
│  │  │  ·  ← 切弧方向（向内侧）      · │  │
│  │  │  ··························│  │
│  │  └─────────────────────────────────────┘  │
│  └────────────────────────────────────────────┘
│  ────────────────────────────────────────┘
```

**MUJoCo 测试用例设计：**

| 测试场景 | 障碍物位置 | 机器狗位置 | 期望行为 | 关键检查点 |
|----------|-----------|-----------|---------|----------|
| 左侧壁撞 | 左边界+障碍物 | 路中 | 向右绕，绕完回路 | 不冲出边界 |
| 右侧壁撞 | 右边界+障碍物 | 路中 | 向左绕，绕完回路 | 不冲出边界 |
| 两侧夹击 | 左右边界+两侧障碍 | 路中 | 减速停止/警报 | 不强行穿越 |
| 序列障碍 | 路右侧序列障碍 | 路中 | 持续向左绕 | 绕完能回路 |
| 盲角突发 | 左边界视觉盲区 | 路右 | 检测到后立即停止 | 不冲出边界 |
| **复杂地面** | 斜坡+障碍物 | 斜坡中段 | 减速绕行 | 姿态稳定 |

## 关键技术决策记录

### 2026-04-19：四足 vs 双足 IMU 参数

四足晃动幅度**小于**双足，频率约为双足 2 倍。因此 glio_mapping 的 IMU 参数（acc_cov、LASER_POINT_COV 等）**先保留原始值**，实测后再调。四足和双足在这块区别不大，不需要激进调整。

### 2026-04-19：视觉避障解耦原因

Marathongo 视觉分割与避障决策当前解耦，是因为半马比赛有专用赛道且逐个出发，只需检测道路和障碍，不需要实时避让行人/其他机器人。四足开放环境导航**必须**补上视觉→避障桥接。

### 2026-04-19：不需要 local_planner

local_planner 是竞速策略，地图导航是通勤逻辑。vo_navigation 的"跟着走+遇障绕行"更匹配。

### 2026-04-19：算力分层 = ABot-Claw 四层

RPA 算力分层（云→机载GPU→YOLO→RL）与 ABot-Claw 架构完全对齐。Hermes 的定位就是"Claw"——云端大模型与端侧运控之间的 Agent 层。这不是独立项目，是同一个系统的不同层级。

### 2026-04-20：导航-避障-运控三层解耦

用户明确提出三层解耦，比 Marathongo 的"导航避控一体"更清晰：
- 导航层只管"往哪走"（waypoints 跟踪）
- 避障层只管"能不能直走"（可行域+绕行）
- 运控层只管"怎么迈步"（cmd_vel → 关节力矩）

三层通过 cmd_vel 交互，彼此不感知内部逻辑。仿真中每层可独立验证。

### 2026-04-20：双引擎仿真策略

用户明确提出**双引擎仿真**：
- **Gazebo Ignition** → 长距离导航（公里级，宏观路径跟踪）
- **MuJoCo** → 短距离避障（百米级，微观运动控制+传感器模拟）
- **不需要同步**，各自独立闭环验证

这是务实的开发策略，避免用一个引擎兼顾所有场景导致的复杂度爆炸。

### 2026-04-20：仿真优先，真机解耦

用户明确要求**开发在仿真中闭环完成，真机只在最后集成阶段参与**。原因：
- 真机测试成本高（电池、场地、机器狗损耗）
- 危险场景（冲出赛道、两侧夹击）在仿真中可安全复现
- 算法迭代速度：仿真秒级 vs 真机小时级
- 机器狗晃动、传感器噪声可在仿真中精确控制

APP后端先简化，用文本化接口替代（JSON/CSV waypoints + 终端日志 + Hermes AI闭环调参）。
