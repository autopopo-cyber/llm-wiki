# Hermes Plan 机制调研与 OpenClaw 对比

> **调研目标**：用户要求每轮对话提炼指令、生成 plan、执行前询问、循环执行、完成总结，多 plan 冲突时修改。调研 OpenClaw 和 Hermes 的 plan 机制，选择最优方案。

---

## 一、OpenClaw 的 Plan 机制

OpenClaw 是一个开源 Agent 框架（GitHub: openclaw/openclaw，36万+ stars），核心设计是**"Plan → Execute → Observe → Reflect → Re-plan" 循环**。

### 1.1 核心流程

```
用户指令
    ↓
Task Decomposition（任务分解）
    ↓
Plan Generation（计划生成）
    ↓
Step-by-Step Execution（逐步执行）
    ↓
Observation & Reflection（观察与反思）
    ↓
Re-planning（重新规划）
    ↓
完成 / 继续执行
```

### 1.2 关键特性

| 特性 | OpenClaw 实现 | 说明 |
|------|--------------|------|
| **任务分解** | 自动分解复杂任务为子任务 | LLM 驱动的递归分解 |
| **计划持久化** | 计划保存为 JSON/YAML | 可中断、可恢复 |
| **执行监控** | 每步执行后检查状态 | 成功/失败/异常三种状态 |
| **动态重规划** | 失败时自动重新规划 | 不依赖初始计划完美 |
| **工具集成** | 计划可包含工具调用 | 每个 step 可绑定 tool |
| **记忆上下文** | 执行历史自动注入上下文 | 支持长程任务 |

### 1.3 ABot-Claw 对 OpenClaw 的扩展

ABot-Claw 基于 OpenClaw 运行时，增加了具身智能特有的能力：

- **VLAC 闭环**：执行后由 Critic 评判，不达标则重规划
- **跨本体路由**：计划步骤可分发到不同机器人
- **空间记忆注入**：计划执行时可检索地图记忆

## 二、Hermes 内置的 Plan 机制

Hermes 已有两个相关 skill，组合起来形成完整的 plan 工作流：

### 2.1 `plan` Skill（计划模式）

**文件**：`~/.hermes/skills/software-development/plan/SKILL.md`

```
用途：用户想要 plan 而不是执行时使用
约束：
  - 不实现代码
  - 不编辑项目文件（除了 plan markdown）
  - 不执行破坏性命令
输出：.hermes/plans/YYYY-MM-DD_HHMMSS-<slug>.md
```

**核心行为**：
1. 检查上下文
2. 写 markdown plan（包含 Goal、Context、Approach、Steps、Files、Tests、Risks）
3. 保存到 `.hermes/plans/` 目录
4. **不执行**

### 2.2 `writing-plans` Skill（写实现计划）

**文件**：`~/.hermes/skills/software-development/writing-plans/SKILL.md`

```
用途：有 spec 或需求时，创建详细的实现计划
粒度：每个任务 2-5 分钟
内容：精确文件路径、完整代码示例、验证步骤
配合：subagent-driven-development（分任务执行）
```

**核心原则**：
- DRY / YAGNI / TDD
- 精确文件路径
- 完整可复制的代码
- 明确的验证步骤

### 2.3 组合工作流

```
用户需求
    ↓
/writing-plans → 生成详细实现计划
    ↓
保存到 .hermes/plans/
    ↓
用户确认 → /subagent-driven-development
    ↓
逐个任务执行 + 两阶段评审
    ↓
完成
```

## 三、OpenClaw vs Hermes 对比

| 维度 | OpenClaw | Hermes (plan + writing-plans) |
|------|----------|-------------------------------|
| **触发方式** | 自动触发（接收指令即分解） | 显式触发（用户说"做 plan"或加载 skill） |
| **计划粒度** | 动态自适应 | 固定 2-5 分钟/任务 |
| **执行控制** | 全自动执行，可中断 | 需显式确认后执行 |
| **用户确认** | 可选（可配置） | **强制**（plan 不执行，需用户说"执行"） |
| **失败处理** | 自动重规划 | 需人工介入或子 agent 失败报告 |
| **记忆集成** | 内置长短期记忆 | 依赖 Hindsight / 会话上下文 |
| **工具调用** | 计划 step 绑定 tool | 子 agent 自主决定 tool |
| **多 plan 冲突** | 运行时队列管理 | **无内置机制**（需自建） |
| **持久化** | JSON/YAML 计划文件 | Markdown 计划文件 |
| **适用场景** | 自动化 Agent、长期任务 | 软件工程、人机协作 |

## 四、关键差距分析

### 4.1 Hermes 缺少什么（相对于用户需求）

| 用户需求 | Hermes 现状 | 差距 |
|----------|------------|------|
| 每轮对话**自动**提炼指令生成 plan | 需显式加载 skill | ❌ 不自动 |
| 执行前**必须询问** | plan skill 不执行，但无强制询问 | ⚠️ 半自动 |
| **自动循环**执行直到完成 | 需手动逐轮推进 | ❌ 不自动 |
| 多 plan **冲突检测与修改** | 无内置机制 | ❌ 缺失 |
| 所有 plan **完成后的总结** | 无内置机制 | ❌ 缺失 |

### 4.2 OpenClaw 缺少什么（相对于军警场景）

| 军警需求 | OpenClaw 现状 | 差距 |
|----------|--------------|------|
| **执行前必须人工确认** | 全自动执行 | ❌ 不适合高危操作 |
| 计划精确到文件路径 | 动态生成，不精确 | ⚠️ 需要适配 |
| 与人协作（非全自动） | 偏向自动化 Agent | ❌ 需改造 |

## 五、推荐方案：融合两者 + 增量建设

### 5.1 核心设计原则

```
全自动（OpenClaw） ←──── 距离 ────→ 全手动（传统开发）
                          ↑
                    我们的位置："半自动人机协作"
                    
- 计划生成：自动（AI 提炼）
- 执行确认：强制人工（军警要求）
- 执行过程：自动循环
- 异常处理：人工介入
```

### 5.2 建议的实现方案

**不改 soul.md**（soul.md 是用户定义的身份层，由用户维护）  
**改 persona.md**（行为层，注入 plan 行为准则）  
**新建 skill**（具体流程，可显式加载）

#### Step 1: 修改 `persona.md`（行为准则注入）

在 persona.md 中增加核心行为指令：

```markdown
## Plan-Execute 工作流准则

每轮用户对话，按以下流程处理：

1. **指令提炼**：提取用户的明确指令和隐含需求
2. **Plan 生成**：若指令可分解为 2+ 步骤，生成 plan
3. **执行前询问**：向用户展示 plan，等待确认
4. **循环执行**：确认后自动执行步骤，每步报告状态
5. **冲突检测**：若有新指令与当前 plan 冲突，暂停并询问
6. **完成总结**：所有步骤完成后，输出执行摘要

约束：
- 不自动执行任何 plan，必须用户确认
- 每步执行后简要报告（成功/失败/阻塞）
- 新 plan 与旧 plan 冲突时，明确告知用户并请求决策
```

#### Step 2: 新建 `plan-execution-workflow` Skill

具体实现流程：

```
用户输入
    ↓
[Intent Parser] 提炼指令
    ↓
是否存在活跃 plan？
    ├─ 是 → 新指令是否冲突？
    │       ├─ 冲突 → 暂停旧 plan，询问用户
    │       └─ 不冲突 → 追加到当前 plan
    └─ 否 → 生成新 plan
    ↓
向用户展示 plan（目标、步骤、预估时间）
    ↓
等待用户确认（"执行"/"修改"/"取消"）
    ↓
Step-by-Step 执行
    ├─ 每步完成 → 简要报告
    ├─ 步骤失败 → 标记阻塞，询问用户
    └─ 外部中断 → 保存状态，等待恢复
    ↓
所有步骤完成 → 生成总结报告
```

#### Step 3: 与现有 `plan` + `writing-plans` Skill 的衔接

| 场景 | 使用 Skill | 输出 |
|------|-----------|------|
| 用户说"plan一下" | `plan` | `.hermes/plans/*.md`（不执行） |
| 用户说"写实现计划" | `writing-plans` | 详细开发计划（不执行） |
| 用户给指令，需要自动 plan-execute | `plan-execution-workflow` | 提炼 → 确认 → 执行 → 总结 |

三者是互补关系，不是替代关系。

### 5.3 与 OpenClaw 的借鉴点

| OpenClaw 特性 | 如何借鉴到 Hermes |
|--------------|-------------------|
| 任务自动分解 | Plan 生成时自动判断是否需要分解 |
| 计划持久化（JSON） | 除 markdown 外，可选 JSON 格式供程序解析 |
| 执行状态机（成功/失败/异常） | 每步报告用统一状态标签 |
| 动态重规划 | 失败步骤不终止，询问用户是否重试/跳过/修改 |
| 工具绑定 | 子 agent 执行时自动选择 tool |

## 六、直接回答用户问题

### Q1: 是否需要改 soul.md 或别的文件？

**答：不需要改 soul.md。**

- `soul.md` 是用户定义的**身份层**（我是谁、我的价值观），由用户维护
- `persona.md` 是**行为层**（我怎么做事、我的风格），可写入 plan 行为准则
- **Skill** 是**流程层**（具体步骤、检查清单），新建 skill 最灵活

推荐路径：
1. **改 `persona.md`** → 注入 "plan-before-action" 核心行为
2. **新建 `plan-execution-workflow` skill** → 具体流程、冲突处理、状态机
3. **不改 `soul.md`** → 保持用户定义的身份不变

### Q2: Hermes 有没有类似 OpenClaw 的 plan 机制？

**答：有，但不完整。**

- `plan` + `writing-plans` 提供了**计划生成**能力
- 但缺少：**自动触发**、**执行前强制确认**、**自动循环执行**、**多 plan 冲突管理**、**完成总结**
- 这些需要增量建设，不是 Hermes 原生支持的

### Q3: 如果没有更好的方案就按之前说的办？

**答：有融合方案，比单纯用 Hermes 原生或模仿 OpenClaw 更好。**

推荐采用 **"Hermes 原生 skill + OpenClaw 借鉴 + 军警确认约束"** 的混合方案：

| 来源 | 采用什么 | 为什么 |
|------|---------|--------|
| Hermes 原生 | `plan` + `writing-plans` skill | 成熟、与 subagent 配合好 |
| OpenClaw 借鉴 | 任务自动分解、状态机、重规划 | 弥补 Hermes 自动化的不足 |
| 军警约束 | 强制人工确认、不可自动执行 | 高危操作必须人在回路 |
