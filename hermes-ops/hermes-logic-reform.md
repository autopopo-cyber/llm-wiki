# Hermes 逻辑变革分析

## 当前 Hermes 逻辑

```
用户发消息 → agent 开始长会话 → 连续调用 N 个工具 → 所有结果攒在内存 → 会话结束时写日志 → 汇报结果
```

**问题：**
1. 会话中途崩溃 → 所有进度丢失（日志只在末尾写入）
2. 单次会话做太多事 → 上下文爆炸、超时、截断
3. 工具调用失败 → 整个会话可能中断
4. 用户中断 → 做到一半的事情没有持久化
5. 无法并行 — 一个 agent 一次只能做一个长会话

## 用户设想的逻辑（"一件事一会话"）

```
循环 plan-tree:
  取下一个最高优先级 plan node
  如果 node 可分解 → 生成子 node，标记当前 node，结束会话
  如果 node 可执行 → 执行这一个 node → 每步增量写日志 → 完成后更新 plan-tree → 结束会话 → 汇报
  等待 ~1min（用户可打断）
  继续下一个 node
```

**优点：**
1. 崩溃恢复 — 每个会话只做一件事，即使崩溃也只丢失当前这一步
2. 增量持久化 — 每步写日志，不攒到最后
3. 清晰进度 — plan-tree 就是进度表，每个 node 状态一目了然
4. 用户可干预 — 每个会话之间有窗口期
5. 可并行 — 多个 agent 各取各的 node
6. 上下文可控 — 每个会话只加载当前 node 需要的上下文
7. 天然的心跳 — 每个会话完成 = 一次心跳

**缺点/挑战：**
1. 会话启动开销 — 每次 cold start 需要重新加载上下文（skill, plan-tree, memory）
2. 上下文切换成本 — 相关任务被拆成多个会话，前一会话的中间结论需要持久化才能传递
3. 粒度问题 — "一件事"怎么定义？太大 → 回到老问题；太小 → 开销过大
4. 依赖关系 — node B 依赖 node A 的输出，A 没完成 B 不能开始
5. LLM token 浪费 — 每个会话都要读 plan-tree + 加载 skill，重复消耗

## 关键洞察

**会话日志丢失的根因不是"日志写太晚"，而是"会话做太多事"。**

如果一个会话只做一件事（5-10 个工具调用），那么：
- 崩溃损失极小（最多重做 5 步）
- 日志可以每步写
- 上下文不会溢出
- 用户随时可以打断

**所以核心变革是：从"长会话做 N 件事"变成"短会话做 1 件事，循环 N 次"。**

## 可行性分析

### 不改 Hermes 源码的方案

利用现有机制实现：
1. **Cron job 每 3-5 分钟触发** → 取 plan-tree 下一个 pending node → 执行 → 写结果 → 结束
2. **每个 node 有独立日志文件** → `~/.hermes/node-logs/<node-id>.md`
3. **Plan-tree 就是任务队列** → 不需要额外的管理系统
4. **3 步完成法** → 执行 node → 更新 plan-tree → 写 node-log → 结束

**优点**: 不需要改 Hermes 源码
**缺点**: Cron 冷启动开销；依赖外部调度

### 改 Hermes 源码的方案

修改 `agent loop`：
1. **增量日志** — 每个工具调用完成后立即 append 到日志文件
2. **单任务会话** — 在 config 里设 `max_tasks_per_session: 1`
3. **自动 plan-tree 循环** — 一个任务完成后自动取下一个，而不是等用户
4. **会话间上下文传递** — 前一会话的摘要自动注入下一会话

**优点**: 从根本上解决问题
**缺点**: 需要改 Hermes 核心代码

### 推荐方案

**先在不改源码的情况下验证概念（方案A），再用丞相测试，满意后改源码（方案B）。**
