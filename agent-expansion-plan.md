## LLM 对比测试方案

### 设计原则
- 生产环境实测 > 标准化 benchmark
- 同一任务、同一 prompt、不同 LLM → 对比输出质量
- 一天一次，嵌入日常开发流程

### 测试维度
1. **代码生成** — 写一个 ROS2 节点 / Hermes skill
2. **架构设计** — 分析 ABot-Claw 源码结构
3. **Debug** — 给一个报错让 LLM 定位原因
4. **长链推理** — 多步规划（如 ISAAC 仿真部署步骤）
5. **指令遵从** — 复杂约束下的输出

### 执行方式
- 给始皇和骠骑分配同一任务（不同 LLM）
- 始皇用 kimi-k2.6，骠骑用 glm-5.1，丞相用 deepseek-v3.2
- 对比标准：正确性、完整性、指令遵从度、耗时
- 结果记录到 ~/llm-wiki/llm-benchmarks/YYYY-MM-DD.md

### 可用模型（通过 OpenRouter）
- glm-5.1 (免费/便宜)
- deepseek-v3.2 (便宜)
- kimi-k2.6 (便宜)
- claude-sonnet-4 (贵，测试用)
- gpt-4o (贵，测试用)

## 多 Agent 扩展评估

### 机器人开发路线需求
1. **仿真环境** — Isaac Sim (需 GPU) → 骠骑负责
2. **导航算法** — Marathongo 研究 + 开发 → 始皇/骠骑
3. **RPA 系统** — 端侧视频流+playbook → 丞相
4. **硬件接口** — Unitree SDK + ROS2 → 等硬件到货

### 扩展建议

#### Phase 1: 当前 4 节点足够
- 1 协调者 + 2 执行者(GPU) + 1 制度师 = 覆盖所有角色
- 性能过剩，暂不需要新增 venv/agent

#### Phase 2: A2/G1 到货后扩展
- 每条机器狗一个 agent（嵌入式部署）
- 新增「探路者」角色 — ISAAC 仿真验证
- 新增「斥候」角色 — 实机 ROS2 控制

#### Phase 3: 分布式协作成熟后
- 按需扩展，Mission-Control 管理
- 多 GPU 推理服务（本地 vLLM）

### 结论
**当前 4 节点足够，不急于扩展。** 性能过剩但任务还没到需要并行的阶段。等 Mission-Control 部署完成、A2A 流程验证后，再根据实际瓶颈决定扩展。
