# 上下文衰减与分层记忆架构

> 2026-04-23 | 架构设计 | 跨模型通用原则
> 关联：[[gbrain-深度分析]]、[[deep-dive-abot-claw]]、[[Claw的启示-从Agent到具身智能的架构统一]]

## 核心问题

所有当前 LLM 都存在**上下文衰减**：随着对话轮次增加、工具输出积累，模型输出质量下降。表现为：偏离任务意图、重复已做工作、遗漏关键约束、幻觉增加。

这不是 bug，是信息论的必然结果——有限信道中信号与噪声的博弈。

## 衰减曲线因模型而异

| 模型 | 体验观察 | 衰减特征 |
|---|---|---|
| GLM-5.1 | ~5 轮后走样明显 | 详细工具输出快速污染上下文 |
| Claude Sonnet | ~10-15 轮可维持 | 长上下文理解较好，但细节仍会丢失 |
| DeepSeek V3 | ~8-10 轮 | 推理密度高，但中间步骤累积仍致衰减 |
| GPT-4o | ~10-12 轮 | 综合能力均衡，衰减较缓 |

**关键**：衰减必然发生，但曲线形状取决于模型架构、训练数据、注意力机制。架构设计不能绑死某个模型的曲线，但必须为衰减的存在做适配。

**设计原则**：假设最坏情况（5 轮衰减），但允许更好模型走更远。即——系统在 5 轮时强制保护上下文质量，如果模型本身能撑 15 轮，保护措施只是多此一举但无害。

## 从物理世界理解衰减的普遍性

机器人系统的信号处理面临同样的衰减问题：

| 领域 | 衰减现象 | 应对策略 |
|---|---|---|
| IMU 测量 | 误差随时间累积（漂移） | 零偏校准 + ESEKF 融合校正 |
| LiDAR 点云 | 长距离点稀疏、噪声大 | 多帧累积 + 去畸变 |
| GNSS 定位 | 多径效应、遮挡丢信号 | 多源融合（IMU+LiDAR+GNSS） |
| Agent 上下文 | 历史噪声累积淹没意图 | 分层记忆 + Prune + 多 Agent |

**共同规律**：单一信源随时间必然退化，只有多源融合 + 定期校准才能维持质量。

Agent 的"多源融合"就是多 Agent 分工——每个 Agent 用新鲜上下文做一部分工作，结果结构化传递。

## 分层记忆：频率分离

不同层级的信息有不同的时间常数。混在一起就是互相干扰。

| 层 | 频率 | 内容 | 生命周期 | 存储 | 衰减对策 |
|---|---|---|---|---|---|
| L0 工作记忆 | 实时 | 当前上下文窗口 | 毫秒 | 200k tokens (volatile) | Prune 旧噪声 |
| L1 会话记忆 | 秒级 | 对话意图+关键结论 | 会话内 | Hindsight recall | 滚动摘要 |
| L2 任务记忆 | 分钟级 | 任务状态、checkpoint | 任务周期 | 持久化 job queue | 冷启动恢复 |
| L3 空间记忆 | 分钟/小时 | 环境状态、障碍物 | 到下次变化 | 覆盖更新 + TTL | 状态覆盖，非追加 |
| L4 技能记忆 | 永久 | 流程、规则、知识 | 不变 | wiki/skill 文件 | git 同步 |

**核心原则**：每一层只保留该频率的信号，其他频率不进来。

### L3 空间记忆 vs L4 知识记忆的关键区别

这是 GBrain 的 Compiled Truth 设计没有区分的：

- **知识记忆**（GBrain 的 Compiled Truth）：我对某人的认知随时间精确化 → **追加证据，逐步修订**
- **空间记忆**（机器人的环境状态）：走廊当前是否畅通 → **最新感知直接覆盖，旧数据是错误数据**

GBrain 混为一谈是因为它只面向知识管理。物理世界必须区分。

## 多 Agent 协作 = 频率分区

### 为什么分布式不是"算力不够才分散"

一个 200k Agent 做复杂任务的衰减问题，不是换更大模型能解决的（更大模型只是衰减更慢，不会消失）。真正的解法是**不让一个 Agent 承载所有频率的信号**。

```
云端研究员（低频，规划层）
  → 上下文全给理解+决策
  → 输出 Playbook（结构化任务序列）
  → 不做执行细节 → 没有高频噪声

机载工程师（中频，编排层）
  → 上下文全给当前任务+工具调用
  → 输出动作指令
  → 不做环境建模 → 没有低频噪声

机载运控（高频，执行层）
  → 上下文全给实时感知+RL推理
  → 输出关节力矩
  → 不做任何规划 → 没有其他频率噪声
```

**每个 Agent 的上下文里只有自己频率的信号。** 分布式的真正意义是信噪比分区。

### 与 ABot-Claw 的三层架构映射

```
ABot-Claw:     Infrastructure → Runtime Core → Memory & Knowledge
我们:          执行层(RL/运控) → Agent层(Hermes) → 理解层(云端LLM)
频率:          高频(1000Hz)    → 中频(0.1-1Hz)  → 低频(0.01Hz)
上下文内容:    关节+IMU       → 任务+工具       → 策略+知识
```

ABot-Claw 的 VLAC 闭环本质上就是在做跨层信号校准——Critic 检查 Action 结果是否符合 Vision+Language 的预期，相当于 Agent 层向理解层做了一次零偏校准。

## 对 Hindsight 增强的具体设计

### 1. retain 按层分类 + TTL

```python
# 现状：所有 retain 都一样
hindsight_retain("某走廊畅通")

# 增强：按层标记 + 生命周期
hindsight_retain("走廊A当前畅通", layer="spatial", ttl="5m")
hindsight_retain("导航决策：走走廊A", layer="task", ttl="task_end")
hindsight_retain("用户偏好用OpenRouter", layer="skill", ttl=None)
```

**TTL 是架构必需，不是可选功能。** 没有垃圾回收的记忆系统 = 没有垃圾回收的内存。

### 2. recall 按层检索 + 权重

```python
# 现状：纯语义相似度
hindsight_recall("走廊状况")

# 增强：多维权重
hindsight_recall("走廊状况", 
    weights={
        "semantic": 0.3,    # 语义相似
        "temporal": 0.3,    # 时间局部（最近的优先）
        "spatial": 0.2,     # 空间邻近（我附近的优先）
        "task": 0.2,        # 任务相关（当前任务需要的优先）
    })
```

向量相似度只是其中一维。物理世界的检索必须考虑空间、时间、任务上下文。

### 3. 持久化 job queue（L2 任务记忆）

```
设计原则：
- SQLite 本地持久化（机器狗端，离线可用）
- Postgres 云端聚合（多机协调）
- 任务 = 结构化数据，不依赖 Agent 上下文
- 失败恢复 = 读 DB 重放未完成任务
- Agent 重启 = 冷启动，从 job queue 恢复状态
```

### 4. delegate_task 作为核心工作模式

任何超过当前模型最佳轮次的工作，自动拆成子任务：
- 每个 subagent 只做 3-N 轮（N 取决于模型衰减曲线）
- 输出结构化结果后立即释放上下文
- 父 Agent 只做调度和审查，不深入细节

不需要改代码，是工作模式的改变。

## 切分 vs 不切分：衰减与耦合的博弈

上文的频率分区和 delegate_task 强调了"切分"的收益，但切分本身有代价：**跨子任务的认知耦合被切断**。

显式依赖（A 的输出是 B 的输入）通过接口定义可以传递。但隐式依赖（A 的中间发现改变了 B 的假设）在切分时必然丢失。

**三档决策**：

| 情况 | 策略 | 原因 |
|---|---|---|
| 任务天然独立 | 切 | 信噪比收益大，切分代价为零 |
| 任务有浅耦合 | 切，但接口要设计好 | 代价可控，收益仍正 |
| 任务有深度耦合 | 不切，换大模型 | 切分的代价 > 衰减的代价 |

**难点在第三档的判断**——什么算"深度耦合"没有客观标准。一个跨三模块的 bug，拆成三个 subagent 各看一个模块，没人能发现 A→B→C 的因果链。一个大上下文模型一次看完所有代码，可能第三轮就发现了。

**实践原则**：
- 先尝试切分，如果 subagent 输出反复需要人工补上下文才能理解，说明耦合太深
- 判断不准时宁可先不切，用完整个上下文跑一轮看效果，再决定是否拆
- 换大模型（1M 上下文）不是偷懒，是对深度耦合问题的正确响应
- 这是个需要反复实践的细活儿，没有一刀切的规则

## 模型无关的设计原则

1. **衰减必然发生**，曲线因模型而异——架构不绑死具体轮次
2. **按最坏情况保护，按最好情况利用**——5 轮保护无害于 15 轮模型
3. **分层是频率分离**——每层只留自己频率的信号
4. **分布式是信噪比分区**——不是算力不够才分散
5. **TTL 是垃圾回收**——没有过期机制的记忆系统必然膨胀
6. **覆盖更新 vs 追加**取决于信息类型——物理状态覆盖，知识证据追加
7. **跨 Agent 通信必须结构化**——不能用自然语言传递任务状态（噪声太大）

## GBrain 的位置

GBrain 验证了：Compiled Truth + Timeline 分层、Self-Wiring Graph、Minions 持久化队列、Skillify 质量体系都是可行的设计。但它的假设（单用户、知识管理、Postgres 中心化、不区分空间/知识记忆）与物理世界不匹配。

**我们借鉴设计洞察，用自己的技术栈实现，针对物理世界的需求做调整。不移植代码，不加双系统。**
