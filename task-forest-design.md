# 仙秦帝国 · 任务森林设计

> 需求来源：秦剑（始皇帝本尊）

---

## 1. 核心概念

### Plan-Tree vs Task-Forest

| 维度 | Plan-Tree | Task-Forest |
|------|-----------|-------------|
| 所有者 | 单个 agent | 协调者 |
| 内容 | 自身的任务规划 | 所有下属的任务状态 |
| 粒度 | 子任务分解 | 执行者+时间线 |
| 用途 | 自我调度 | 指挥调度 |
| 关系 | 个人日程 | 团队看板 |

### Task-Forest 节点结构

```
├── 任务名称
│   ├── 执行者: [始皇/骠骑/丞相]
│   ├── 状态: [待分配/进行中/已完成/超时/失败]
│   ├── 开始时间: 2026-04-24 23:00
│   ├── 预期结束: 2026-04-24 23:30
│   ├── 实际结束: (未完成)
│   ├── 子任务: [...]
│   └── 备注: ...
```

## 2. 心跳机制

### 上报格式
每5分钟各agent通过API上报：
```json
{
  "agent": "shihuang",
  "timestamp": "2026-04-24T23:30:00Z",
  "status": "busy",
  "current_task": "deploy-memos",
  "progress": "70%",
  "eta": "2026-04-24T23:45:00Z"
}
```

### 超时规则
- 预期结束时间 + 10分钟 → 相邦主动问询
- +30分钟仍未响应 → 标记为失败，重新分配
- 连续3次心跳丢失 → 标记agent离线

## 3. 与 Mission-Control 集成

Mission-Control (4333⭐) 提供：
- 可视化 dashboard（Next.js）
- 任务/agent 状态管理（SQLite）
- MCP 协议原生支持
- OpenClaw A2A 兼容

### 集成方案
1. 在始皇或骠骑上部署 Mission-Control
2. 各agent通过 MCP 上报状态
3. 相邦读取 Mission-Control API 获取森林视图
4. 心跳超时检测由 Mission-Control 的 webhook 触发

---

## 4. 协调者工作流

```
1. 接收秦剑的任务指令
2. 分解任务 → 写入 Task-Forest
3. 分配给最合适的执行者
4. 等待心跳上报
5. 预期时间到 → 检查完成情况
6. 超时 → 问询/重新分配
7. 完成 → 更新 Task-Forest → 汇报秦剑
```

---

## 5. 优先级与并行

- 无依赖关系的任务 → 并行分配给不同agent
- 有依赖关系的任务 → 串行，标注关键节点
- 关键节点 = 阻塞其他任务的完成点
- 协调者维护依赖图，自动识别关键路径
