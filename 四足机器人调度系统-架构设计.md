# 四足机器人调度系统 — 架构设计

## 项目概述

基于 Unitree A2（兼容 Go2）四足机器人 + 安卓调度 App + 双服务器后端的完整系统。

## 硬件资源

| 资源 | 配置 | 用途 |
|------|------|------|
| 开发服务器 ×2 | 64G RAM + RTX 2080 Ti 11GB | 仿真训练、VLM推理、路径规划、后端服务 |
| 宇树 A2 ×1 | 兼容 Go2 SDK / ROS2 | 实机验证 |

**2080 Ti 11GB 能跑什么：**
- YOLO11s-seg TensorRT FP16 ✅
- Qwen2.5-VL-3B/7B INT4 ✅（端侧用，服务器可跑更大）
- MuJoCo/Isaac Sim 仿真 ✅
- RL 策略训练 ✅（PPO/MAPPO）
- 深度估计（Depth Anything V2）✅

## 安卓 App 功能模块

### 1. 高德地图 SDK 集成

```kotlin
// 核心功能
- 显示机器人实时位置（MQTT 上报坐标 → GCJ-02 偏移 → 地图标注）
- 长按选点 → 生成导航任务 → 调用高德步行API → 下发路径
- 显示 waypoints 轨迹线
- 显示机器人历史轨迹（热力图/折线）
```

**坐标系处理：**
```
App 显示：GCJ-02（火星坐标，高德原生）
   ↓
云端存储：WGS-84
   ↓
机器人本地：UTM / 以起点为原点的局部坐标（FLU）
```

### 2. 机器人列表

| 字段 | 说明 |
|------|------|
| 在线状态 | MQTT 心跳 5s 超时判定 |
| 电量 | A2 电池状态上报 |
| 当前位置 | SLAM 位姿 → UTM → WGS-84 → GCJ-02 |
| 当前任务 | 任务ID + 进度 |
| 传感器状态 | LiDAR / Camera / GNSS / IMU 健康度 |
| 控制按钮 | 急停 / 召回 / 任务下发 |

### 3. 任务调度系统

**任务类型：**

| 类型 | 说明 | 示例 |
|------|------|------|
| 导航任务 | 点到点导航 | "去图书馆门口" |
| 巡逻任务 | 按 waypoints 循环 | "绕教学楼巡逻3圈" |
| 跟随任务 | 跟随目标（人/车） | "跟随前方人员" |
| 定时任务 | 预约执行 | "每天8点去门口拍照" |
| 协作任务 | 多机配合 | "A狗去东门，B狗去西门" |

**Playbook 格式（JSON）：**

```json
{
  "task_id": "nav_001",
  "type": "navigation",
  "priority": 1,
  "waypoints": [
    {"lat": 39.9896, "lng": 116.4810, "action": "none"},
    {"lat": 39.9900, "lng": 116.4820, "action": "capture"}
  ],
  "constraints": {
    "speed_limit": 1.5,
    "avoid_zones": [[...]],
    "wait_at_goal": 10
  }
}
```

### 4. 实时监控

- 机器人第一视角视频流（WebRTC / RTSP，延迟 <500ms）
- SLAM 轨迹叠加地图
- 传感器数据图表（IMU、电池、信号强度）
- 告警推送（跌倒、失联、低电量、碰撞）

## 通信协议设计

### 机器人←→云端：MQTT

**主题设计：**

```
robot/{robot_id}/heartbeat       → 机器人上报心跳（5s）
robot/{robot_id}/state           → 机器人上报状态（位置/电量/传感器）
robot/{robot_id}/telemetry       → 高频遥测（IMU/速度，10Hz）
robot/{robot_id}/camera          → 视频流（WebRTC 单独通道）
cloud/{robot_id}/task            → 云端下发任务
cloud/{robot_id}/path            → 云端下发路径（waypoints）
cloud/{robot_id}/cmd             → 云端下发即时指令（急停/召回）
```

**消息格式（Protobuf）：**

```protobuf
message RobotState {
  string robot_id = 1;
  double timestamp = 2;
  Pose pose = 3;              // 局部坐标系位姿
  GeoPose geo_pose = 4;       // WGS-84 经纬度高程
  Battery battery = 5;
  repeated SensorStatus sensors = 6;
  string current_task = 7;
}

message Path {
  string task_id = 1;
  repeated Waypoint waypoints = 2;
  double speed_limit = 3;
}
```

### App←→云端：HTTPS + WebSocket

- **HTTPS**：用户认证、机器人列表、历史数据、任务管理
- **WebSocket**：实时推送（机器人位置更新、任务状态、告警）

## 数据流

### 导航任务完整流程

```
1. App 长按地图选点
   ↓
2. App 调用高德步行路径 API
   ↓
3. 云端：坐标转换（GCJ-02 → WGS-84 → UTM → 局部坐标）
   ↓ 云端：路径优化（Subgoal 下采样、转弯检测）
   ↓
4. 云端通过 MQTT 下发 Path 消息给机器人
   ↓
5. 机器人：path_process_node 接收 waypoints
   ↓
6. 机器人：vo_navigation 跟踪路径 + 避障
   ↓
7. 机器人：实时上报位姿 → 云端 → App 显示
   ↓
8. 到达目标 → 上报完成 → App 推送通知
```

## 服务器部署方案

### 双 2080 Ti 分工

| 服务器 | 角色 | 部署内容 |
|--------|------|---------|
| **Server A** | 仿真训练 + VLM推理 | MuJoCo/Isaac Sim、RL训练、Qwen2.5-VL-7B、YOLO训练 |
| **Server B** | 生产后端 | API Gateway、调度服务、MQTT Broker、PostgreSQL、Redis |

**可切换：** 开发阶段两台都跑仿真；部署阶段 Server A 降级为备份。

### Docker 化部署

```yaml
# docker-compose.yml（Server B 生产环境）
services:
  nginx:
    image: nginx:alpine
    ports: ["80:80", "443:443"]
  
  api:
    build: ./backend
    environment:
      - DATABASE_URL=postgresql://...
      - REDIS_URL=redis://redis:6379
      - MQTT_BROKER=mqtt:1883
  
  mqtt:
    image: eclipse-mosquitto:2
    ports: ["1883:1883", "9001:9001"]
  
  postgres:
    image: postgres:15
    volumes: ["pg_data:/var/lib/postgresql/data"]
  
  redis:
    image: redis:7-alpine
```

## 与导航开发路线的衔接

| 开发阶段 | 导航系统 | 调度系统 | 验证方式 |
|---------|---------|---------|---------|
| Phase 1 定位移植 | glio_mapping 参数调优 | — | MuJoCo 仿真 + 实机 |
| Phase 2 极简导航 | vo_navigation 跟踪 | — | MuJoCo waypoints + 障碍物 |
| Phase 3 复杂导航 | local_planner 候选采样 | — | MuJoCo 多场景测试 |
| Phase 4 视觉融合 | YOLO + 避障桥接 | — | MuJoCo 语义障碍物 |
| **Phase 5 地图对接** | **高德 API → waypoints** | **App 路径下发** | **实机校园测试** |
| **Phase 6 调度集成** | **完整导航栈** | **App + 云端后端** | **多机协作测试** |

## 关键技术决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 通信协议 | MQTT + WebSocket | MQTT 弱网友好、断线重连；WebSocket 实时推送 |
| 地图服务 | 高德 SDK（App）+ 高德 API（云端） | 国内合规、步行路径成熟、坐标系原生支持 |
| 视频流 | WebRTC | 低延迟（<500ms）、P2P 打洞、适配移动端 |
| 数据库 | PostgreSQL + PostGIS | 支持地理坐标查询、轨迹存储、时空分析 |
| 机器人 SDK | unitree_sdk2（Go2兼容） | A2 兼容 Go2，ROS2 Humble，社区成熟 |
| 机载计算 | Jetson Orin NX 16GB | DeepStream 7.1、TensorRT、与 A2 生态匹配 |

## 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| 高德 API 调用限流 | 路径规划失败 | 本地缓存常用路径、降级为直线导航 |
| 4G/5G 信号弱 | 机器人失联 | MQTT QoS 1 + 本地缓存路径 + 断线续跑 |
| Jetson 算力不足 | VLM 延迟高 | VLM 降频（1Hz）、YOLO 全帧（30Hz）、关键帧触发 VLM |
| 多台机器人碰撞 | 安全事故 | 云端协调（时空分离）、本地 LiDAR 紧急制动 |
| 坐标系漂移 | 定位偏差 | RTK 校准、回环检测、定期重对齐 |

## 与导航/避障/跟随/编队的衔接

| 系统 | 输入 | 输出 | 协同方式 |
|------|------|------|---------|
| **导航** | waypoints（高德/缓存） | 目标航向+速度 | Phase 1-4 核心开发 |
| **避障** | LiDAR + 视觉 + 可行域 | 修正 cmd_vel | 始终在后台运行 |
| **人跟随** | 视觉(YOLO+ReID) + LiDAR(腿模式) | 跟随目标位姿 | 本地闭环，不强制联网 |
| **编队** | UWB 邻居位姿 + 队形定义 | 期望相对位置 | P2P 通信，去中心化 |
| **调度后端** | App 下发任务/更新 | Playbook 版本同步 | 异步，弱网可用 |

**去中心化设计原则：** 跟随和编队的核心算法在机载闭环运行，后端只做任务分发、版本更新、数据收集，不干预实时控制。
