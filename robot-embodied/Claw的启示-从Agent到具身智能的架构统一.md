# Claw 的启示：从 Agent 到具身智能的架构统一

> 当 ABot-Claw 在亦庄半马封神时，它验证的不只是一个机器人框架——它验证的是一个架构原则：**Agent 是 LLM 与物理世界之间的那层胶水。** 而这个原则，和你正在用的 Hermes Agent，是同一个东西。

---

## 一、VLA 的困境：大模型直接控制机器人，为什么失败了？

2024 年，整个具身智能领域都在押注 VLA（Vision-Language-Action）。思路很直觉：大模型理解了图像和语言，让它直接输出机器人的动作指令，端到端，一步到位。

Google DeepMind 的 RT-2 用 55B 参数的模型输出离散动作 token；OpenVLA 用 7B 参数做同样的尝试；Physical Intelligence 的 π₀ 改用 flow matching 替代离散 token。但它们都遇到了同一个问题：

**1. 太慢。** 自回归生成速度 1-5Hz，机器人控制需要 20-50Hz 以上。VLA 为了提速做 action chunking（一次预测多步开环执行），但这又带来了误差累积。

**2. 太粗。** RT-2 把 7 维连续动作量化为 256 个离散 token，每维只有 256 个级别。精细操作需要亚厘米精度，离散 token 力不从心。π₀ 转向连续动作输出，正是对这一问题的承认。

**3. 越走越偏。** 理论证明：直接在动作空间规划时，可行轨迹概率随时间窗口增加而指数衰减。长时域任务中，VLA 不可避免地偏离目标。

**4. 学了控制，忘了理解。** 微调控制数据会损害 VLM 原有的推理能力——端到端训练导致语义理解与运动控制产生根本矛盾。

**5. 开环脆弱。** 为了效率做开环执行，缺少闭环反馈，环境一变就崩。

一句话总结：**VLA 失败，是因为让理解层直接跳到了执行层，跳过了中间那层。**

---

## 二、RL 的胜利：底层运控不需要语言理解

与 VLA 的困境形成鲜明对比，RL 运控在 2024-2025 年大放异彩：

- Unitree G1 人形机器人跳舞、打功夫
- Boston Dynamics Atlas 跑酷
- Figure 01 与 OpenAI 合作实现语音交互+RL 执行
- Tesla Optimus 用 RL 运动控制+视觉感知

RL 运控成功的关键因素很清楚：

| 维度 | VLA 端到端 | RL 运控 |
|------|-----------|---------|
| 控制频率 | 1-5Hz | **50-1000Hz** |
| 动作精度 | 离散 256 级 | **连续浮点精度** |
| 闭环反馈 | 开环或慢闭环 | **天然闭环** |
| 数据需求 | 大量真实演示 | **仿真训练+少量真实数据** |
| 语义理解 | 有但微调后退化 | **不需要，也不该有** |

RL 的成功说明了一个事实：**低层运动控制应该是专门化的，不需要语言理解能力。** 机器人不需要"想"怎么迈步，它只需要"感受"——IMU 反馈、关节角度、足端触地，然后在 10ms 内完成姿态调整。这是脊髓反射，不是大脑皮层。

## 三、Agent 的兴起：LLM 不做控制，做编排

2025-2026 年，Agent 代理范式兴起，找到了 LLM 与具身智能结合的正确方式：

```
旧路线 (VLA)：  VLM → Action Token → 机器人执行       （太慢太粗，开环脆弱）
新路线 (Agent)：LLM → Agent → Playbook → RL Policy   （各层各司其职）
                 "理解"  "编排"   "序列"    "执行"
```

LLM 不再直接输出动作，而是输出**意图**；Agent 把意图翻译成可执行的**步骤序列**（Playbook）；RL 策略精确执行每一步。三层分明，各司其职。

这个范式在不同尺度上重复自身：

```
机器人世界：                    Agent 系统自身：
─────────────                  ─────────────
LLM (理解意图)        ←→       GLM-5.1 (理解对话)
Agent (编排工具)      ←→       DeepSeek v3.2 (调用工具)
RL/OS (精确执行)      ←→       终端/代码/浏览器 (精确操作)
```

不是类比，是同一个架构原则：**理解层只做判断不做执行，Agent 层是桥梁把意图翻译成步骤，执行层只管精确不管为什么。**

## 四、ABot-Claw：这个原则在 3D 时空的验证

高德 ABot-Claw 是阿里 AMAP CV Lab 推出的机器人智能体进化框架，基于 OpenClaw 运行时构建。它的核心论点正是：**VLA 是开环的，Agent 才能闭环。**

### 4.1 ABot 三层飞轮

ABot 不是一个模型，而是一个三层飞轮：

| 层级 | 模型 | 做什么 |
|------|------|--------|
| **数据层** | ABot-World | 14B DiT 世界模型，批量合成训练数据 |
| **模型层** | ABot-N（导航）+ ABot-M（操作） | 导航 5 类任务 7 项 SOTA，操作 LIBERO 98.6% |
| **应用层** | ABot-Claw | VLAC 闭环 + 跨本体记忆 + 多机协作 |

飞轮效应：World 生成数据 → 训练 Model → Claw 部署执行 → 真实运行数据回流 → World 再生成。能力螺旋上升。

### 4.2 三层解耦架构

ABot-Claw 的工程实现是经典的三层解耦：

```
┌─────────────────────────────────────┐
│   OpenClaw 层（大脑 / System 2）      │
│   自然语言理解 → 任务分解 → 代码生成   │
│   "去哪、做什么"                      │
├─────────────────────────────────────┤
│   共享服务层（小脑 / 感知+记忆+评估）   │
│   VLM / YOLO / GraspNet / Memory    │
│   "看到了什么、记住了什么、做得怎样"    │
├─────────────────────────────────────┤
│   机器人层（脊髓 / System 1）          │
│   ROS 驱动 / RL 运控 / 传感器         │
│   "怎么站稳、怎么迈步"                │
└─────────────────────────────────────┘
```

注意这里的对应关系：

| 认知科学 | ABot-Claw | Agent 系统 |
|---------|-----------|-----------|
| 大脑皮层 (System 2) | OpenClaw 层 | LLM 理解层 |
| 小脑 (感知+协调) | 共享服务层 | Agent 工具层 |
| 脊髓反射 (System 1) | 机器人层 | 代码/终端执行层 |

### 4.3 四大发明：Agent 在 3D 时空需要的工具

ABot-Claw 的核心贡献，是把 Agent 在文本域的工具搬到了 3D 时空：

| Agent 文本域工具 | ABot-Claw 3D 时空工具 | 解决的问题 |
|-----------------|---------------------|-----------|
| 工具返回值 | **Critic（VLAC 的 C）** | 怎么知道动作做没做对 |
| 上下文记忆 | **跨本体多模态记忆** | 怎么记住去过哪、见过什么 |
| delegate_task | **弹性多智能体调度** | 怎么让多个机器人协作 |
| 文件系统/浏览器 | **Map as Memory** | 怎么表示和理解空间 |

逐一解释：

**Critic — 3D 世界的工具返回值**

在文本 Agent 中，工具调用后返回文本结果，Agent 据此决定下一步。在 3D 世界中，机器人执行了一个动作，谁来告诉 Agent 结果如何？ABot-Claw 用 Generalist Reward Model 做在线评估，返回评分信号：

- 分数超过阈值 → 任务完成，进入下一步
- 分数在改善中 → 保持当前策略，局部调整
- 分数停滞或下降 → 当前策略无效，触发重规划

这就是 VLAC 中 "C" 的含义——Vision-Language-**Action-Critic**，从开环变成闭环。

**跨本体多模态记忆 — 3D 世界的上下文记忆**

文本 Agent 的上下文窗口有限，但可以用 RAG 检索长期记忆。3D 世界中，机器人需要记住空间信息——这个房间有什么物体、走廊尽头是什么、昨天探索过的路线。

ABot-Claw 构建了四层视觉空间记忆：图像语义记忆（场景级特征）、关键帧记忆（稀疏关键视角）、物体中心记忆（物体级特征+位置）、地点锚点记忆（空间锚定）。关键创新是**跨本体共享**——A 机器人探索过的环境，B 机器人零成本继承。

**弹性多智能体调度 — 3D 世界的 delegate_task**

文本 Agent 可以 spawn 子任务并行执行。3D 世界中，一条狗去检查电梯、一条狗去迎接访客、一条机械臂整理桌面——同一个运行时，多个机器人并行。

ABot-Claw 的"一个运行时、多智能体共生"设计：动态路由表跟踪每个机器人的能力和状态，任务分解后按能力分派，支持串行/并行/混合协作。一条狗断线了，运行时派另一条去查看——这就是技术报告中的跨本体任务移交演示。

**Map as Memory — 3D 世界的文件系统**

文本 Agent 用文件系统组织和检索信息。3D 世界中，空间就是信息载体。ABot-Claw 用几何地图+语义地图+图像特征索引，构建了以全局空间坐标为锚点的"空间文件系统"。查询"厨房在哪"不是搜索文本，而是检索地点锚点；"橙色物体"不是搜索标签，而是检索物体记忆。

## 五、从导航系统看 Agent 的落地

ABot-Claw 验证了上层架构，但具体到机器人"怎么走"，需要底层导航系统支撑。朗毅机器人开源的 Marathongo 项目提供了一个完整的工程实例。

### 5.1 五层算力栈

导航系统的算力需求天然分层，与 ABot-Claw 的三层架构完美对齐：

| 层级 | 频率 | 任务 | 对应 ABot-Claw |
|------|------|------|---------------|
| L4 云端 LLM/VLM | 0.1-1Hz | 理解意图、生成路线 | OpenClaw 层 |
| L3 机载 VLM | 1-5Hz | 场景理解、语义判断 | 共享服务层 |
| L2 机载 YOLO | 10-30Hz | 目标检测、避障感知 | 共享服务层 |
| L1 RL 运控 | 100-1000Hz | 平衡、步态、力矩输出 | 机器人层 |
| L0 OS/驱动 | 1000Hz+ | 传感器采样、Can 总线 | 机器人层 |

注意频率的指数级递增：云端 0.1Hz，端侧 1000Hz，差四个数量级。没有任何单一模型能覆盖这个范围，必须分层。

### 5.2 导航系统的核心：IMU-LiDAR 融合去畸变

Marathongo 最有价值的技术是 glio_mapping 模块——GNSS+IMU+LiDAR 紧耦合融合定位。核心是 IMU-LiDAR 融合去畸变：

机器人运动时，激光雷达一帧扫描内机器人本身也在运动，点云会产生畸变。去畸变的原理是：IMU 以 200Hz+ 采样记录机器人每个时刻的位姿，激光点云逐点查找该时刻的 IMU 位姿，反向补偿到帧末坐标系。

这个过程的数学本质是在流形上做误差状态卡尔曼滤波（ESEKF），状态空间是 23 维：位置(3)+姿态(3)+外参旋转(3)+外参平移(3)+速度(3)+陀螺零偏(3)+加计零偏(3)+重力方向(2)，其中姿态和外参旋转在 SO(3) 流形上，重力方向在 S² 流形上。

后端用 GTSAM ISAM2 做增量式位姿图优化：里程计因子 + GPS 因子 + 高度因子，关键帧阈值 0.3m+0.1rad。

### 5.3 避障：切弧导航的优雅

Marathongo 的 vo_navigation 模块实现了一种优雅的避障策略——切弧导航：

1. 前方梯形区域检测障碍物（速度越快梯形越宽）
2. 对每个障碍物模拟不同角速度下的圆弧轨迹碰撞
3. 取碰撞圆弧的切线方向作为避障航向
4. 左右两侧分别搜索，选最接近目标方向的航向
5. 速度规划：最近障碍距离→目标速度→低通滤波+加速度限制

本质是"跟线+绕障"——沿着全局路径走，遇到挡路的临时偏移，绕完回来继续跟线。这正是日常导航软件的逻辑，也最适合四足机器狗的通勤场景。

### 5.4 对接地图：从预录路径到动态分发

Marathongo 的路线全部是预录的——要么是人遥控走一遍录下来的轨迹，要么是用数学公式批量生成的参考曲线族。这在马拉松比赛场景够用，但在日常长距离导航中不可行。

ABot-Claw 解决了这个问题的上层：路线由云端动态分发。高德地图 API 生成步行路径（经纬度序列），云端做坐标转换（GCJ-02→WGS84→局部坐标系）和 Subgoal 下采样，然后下发到端侧导航系统执行。

```
高德 API → 经纬度路径 → 坐标转换 → Subgoal 下采样 → vo_navigation → cmd_vel → 机器狗 RL 运控
```

这样导航系统只需暴露"接收路径"的接口，不关心路径从哪来。Marathongo 提供了底层"怎么走"，ABot-Claw 提供了上层"去哪"。

## 六、Claw = Hermes：同一个架构，不同的世界

回到开头的问题：为什么叫 Claw？

Claw 是"爪"，是抓取动作，是 Agent 把云端意图"抓"到端侧执行的那只手。在 ABot-Claw 中，它把 VLM 的语义理解"抓"成 RL 能执行的 Playbook。在 Hermes Agent 中，它把 GLM-5.1 的对话理解"抓"成 DeepSeek 们能调用的工具序列。

同一个架构，不同的世界：

| 维度 | ABot-Claw | Hermes Agent |
|------|-----------|-------------|
| 理解层 | 云端 VLM | GLM-5.1 |
| Agent 层 | OpenClaw + Critic + Memory | DeepSeek v3.2 + Skills |
| 执行层 | RL 运控 + ROS | 终端 + 代码 + 浏览器 |
| 工具返回值 | Critic 评分 | 工具输出 |
| 上下文记忆 | 跨本体多模态记忆 | 对话上下文 + Hindsight |
| 多机协作 | 弹性调度 | delegate_task |
| 空间表示 | Map as Memory | 文件系统 + Obsidian Wiki |

ABot-Claw 造了四个 3D 工具，让 Agent 在物理世界中能"看见结果、记住空间、协调多体、理解环境"。Hermes 天然就有这些能力——在文本域。二者的差距不在架构，而在工具的维度。

**VLA 没有死，它只是换了一种实现方式。** 从"大模型直接输出动作"变成"大模型输出意图，Agent 编排成序列，RL 执行动作"。ABot-Claw 验证了这条路走通了——机械臂能抓取、人形能导航、四足能引导访客，而且能跨本体协作。这不是特例，是架构的力量。

## 七、实践路线：从 Marathongo 到四足导航系统

理论归理论，落地归落地。一个机器狗的室内外长距离导航系统，可以按以下路线搭建：

**Phase 1：定位层移植（1-2 周）**

glio_mapping 配置调整：禁用或重评估 ZUPT（四足步态可能误触发零速检测），调整 IMU 噪声参数，确认 LiDAR-IMU 外参标定。四足晃动幅度小于双足但频率约为双足 2 倍，先保留原始参数实测再调整。

**Phase 2：导航层移植（2-3 周）**

vo_navigation 切弧避障 + PID 控制。核心改造：差速模型假设→全向模型，碰撞体→四足步态包络，运动学参数重新标定。

**Phase 3：地图对接（3-4 周）**

高德步行路径 API → GCJ-02 坐标转换 → Subgoal 下采样（平路 5-10m 间距，弯道 2-5m，楼梯 0.5-1m）→ 导航系统接收路径执行。室内外切换：GNSS 退化检测（卫星数 <6 或 HDOP>4）→ 纯 SLAM 模式 → GNSS 恢复时 EKF 平滑过渡。

**Phase 4：视觉融合（2-3 周）**

YOLO 分割 → ObstacleArray.msg → vo_navigation 消费。Marathongo 的视觉和避障当前是解耦的（半马专用赛道不需要实时避人），四足开放环境必须补上这个桥接。

**Phase 5：RPA 协作对接（4-6 周）**

路线动态分发（云端生成→端侧执行）、任务上下文上云（跨机移交）、空间记忆共享（A 狗探索 B 狗继承）。参考 ABot-Claw 的弹性调度和 Map as Memory，但从轻量级开始——先做单狗+远控，再加多狗协作。

---

## 八、结语：那个被跳过的中间层

2024 年，VLA 试图让大模型直接控制机器人。它失败了，不是因为模型不够强，而是因为**缺了中间那层**。

文本域的 Agent 之所以能跑通，是因为有工具调用——搜索、代码执行、浏览器，这些工具让 Agent 的"意图"能落地。3D 域没有这些工具，大模型就是裸奔。ABot-Claw 的贡献就是造了这些 3D 工具：Critic 让 Agent 能看到结果，Memory 让 Agent 能记住空间，Scheduler 让 Agent 能协调多体，Map 让 Agent 能理解环境。

Claw 的字面意思是"爪"。云端太慢，端侧太笨，中间需要一个 Agent 把意图抓下来、翻译成端侧能执行的序列、把结果再送回去。ABot-Claw 的 Claw 是高德做的。你的 Claw 叫什么？
